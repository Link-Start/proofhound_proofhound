# 24 ┬Ę Experiments

## 1. The role of this page

An experiment is an **offline regression test**: take one prompt version, one dataset, and one model, run all samples once, and produce metrics.

**There is no hard upper limit on the total number of dataset samples**, and an experiment runs all of them by default; the implementation layer never rejects experiment creation based on sample count. To support large datasets, `ExperimentWorkflow` enumerates `dataset_samples` in a streaming fashion via keyset pagination and enqueues them page by page into the `llm` queue, rather than reading all samples into memory at once (avoiding OOM on large datasets). The UI may display the estimated number of calls / duration / cost for very large experiments and require a second confirmation, but once confirmed it must allow creation and must not block based on sample count.

It is the **first quality gate** before a prompt goes liveŌĆöa version whose experiment metrics are acceptable can be promoted to a canary release, or go straight (`from_experiment`) to a production release.

Experiment names are unique within a project; manually created experiments and the baseline / sub-experiments generated by optimizations cannot share a name with any non-deleted experiment in the same project. When a user enters a name, the frontend should warn "This name is already in use" if it detects a duplicate within the project, while the backend still enforces this via a uniqueness constraint as a backstop.

## 2. List page

- Displays all experiments in the instance: name, corresponding prompt version, dataset, model, status, progress, key metrics
- The list page supports a **table view / comparison view**. The comparison view replaces the per-status dashboard columns and is used to compare multiple experiment results side by side: entering the comparison view pre-selects no experiments by default; the top shows a table of the experiments added to the comparison, and the main area shows bar-chart metrics; even with only a single experiment added, its bar chart is shown directly
- The comparison view offers single-experiment add and quick add. Quick add supports batch-adding experiments by prompt, dataset, or optimization task; the optimization-task group includes that task's source experiment (`sourceExperimentId`) and the per-round sub-experiments whose `experiments.optimization_id` points to that task, and the frontend deduplicates by experiment ID when merging into the current comparison list
- Comparison-view metrics fall into two groups: **quality metrics** (Accuracy / Precision / Recall / F1) and **engineering metrics** (P50 / P95 / average latency, Token, cost, failed samples). The frontend prefers the `metrics` snapshot and `failedSamples` from the experiment list DTO
- The comparison view's quality metrics compare on the overall (Overall) basis by default; when the selected experiments' `metrics.perClass` carries classification categories, the quality-metrics area offers a category-dimension selector. After selecting a category, the Precision / Recall / F1 cards switch to that category's per-class metrics across experiments; because the current DTO does not carry per-class TN / FP, Accuracy shows as no data under the category dimension, and the frontend does not derive a pseudo-precision
- Status: **running / success / failed / stopped**
- The dataset column in the list only shows the dataset name + sample count; when the dataset's `hasImages=true` but the prompt version does not declare an image variable, it appends `<UnusedImagesBadge scope="experiment">` (gray outline + `ImageOff` + a tooltip "contains images but unused") as a quick warning
- **Modality icons are concentrated in the "Experiment spec" area on the experiment detail page** (the "Basics" group in the `experiments.detail.spec` block in the right sidebar of the detail page): the prompt row appends prompt-variable modality icons (derived from the version's `variables[].type`, deduplicated; the three image subtypes all render as an image icon), and the dataset row appends the dataset's own modality icon (derived from `datasets.fieldSchema` field roles into text / image). The two icon groups share the `--modality-*` token, with tooltips "Prompt variable: xx" / "Dataset modality: xx" respectively. The list panel keeps information density lean and leaves modality comparison to a one-time check on the detail page
- The backend `ExperimentListItemDto` and `ExperimentDetailDto` additionally carry `promptVariableTypes: PromptVariableType[]` (deduplicated) + `datasetHasImages: boolean` + `datasetModalities: DatasetModality[]` (`'text' | 'image'`, at least one item, falling back to `['text']` on empty / corrupt jsonb); the derivation function `deriveDatasetModalities` is shared between frontend and backend in `packages/shared/dto/dataset-modality.ts`, requiring no N+1 query
- Supports filtering by status and sorting by metric
- Delete / batch delete must first show a second confirmation; after confirmation the backend permanently deletes the experiment as a child work item. This does not modify the upstream prompt or dataset. Owned run results and annotations are deleted as descendants of the experiment deletion plan. If owned run results have offloaded payload shards, the delete path collects only shard refs no longer referenced by any remaining run result and best-effort deletes those `run_result_shard` objects after the DB delete commits; object-storage cleanup failures are logged and do not roll back the experiment deletion. Release events that were created from the experiment continue to display from their release snapshots; the experiment reference is treated as attribution, not lifecycle ownership.
- Batch stop / resume

## 3. Creating an experiment

The creation form is multi-step:

1. Select the prompt version (defaults to the latest version)
2. Select the dataset; the selector only lists completed datasets. A dataset that is still being imported has not yet been written to the `datasets` table and is naturally unselectable (see [22 ┬¦3.1.2](22-datasets.md#312-raw-upload--asynchronous-backend-import) and [22 ┬¦3.1.3](22-datasets.md#313-legacy-client-streamed-batched-import)), so there is no need for an extra dataset-status gate on the experiment side
3. Select the model; only when the selected prompt version declares an image variable (`image` / `image_url` / `image_base64`) does the model selector narrow to image-modality models
4. Configure run parameters: concurrency, **RPM limit**, **TPM limit**, temperature, **retry count** (internal retries within a single sample's LLM call, retrying HTTP 5xx / 408 / 429 / network errors, without affecting BullMQ job-level attempts), per-sample timeout; only when the current model is an image-modality model and the prompt version declares an image variable does it additionally show the image encoding method. The **experiment-level RPM / TPM / concurrency are this experiment's voluntary self-throttling limits**, and before calling the LLM the worker takes the minimum against the model-level quota (`min(model, experiment)`)ŌĆöthe model level is always the ceiling and cannot be raised ([21 ┬¦6](21-models.md#6-rpm--tpm--concurrency)). Experiment-level `concurrency` only tightens the model-level concurrency ceiling; **whether auto-concurrency is used is determined by the model-level `auto_concurrency`, and the experiment does not override it**ŌĆöwhen the model has auto enabled, the experiment-set concurrency is the "self-tuning upper bound" rather than a hard concurrency ([21 ┬¦6.1](21-models.md#61-auto-concurrency))
5. **Alignment validation**: the platform pre-checks before submission whether the prompt template variables are covered by the dataset and whether the image fields match the model's capabilities. When selecting a dataset other than the default binding, it must additionally validate whether that dataset covers all text / numeric / image variables declared by the prompt version; if any variable is missing, a red blocking warning is shown and submitting the experiment task is disallowed. A non-blocking validation failure still lists a warning, and the user can choose to continue or go back to fix it

After submission the experiment immediately enters the "running" status, and the server synchronously starts a DBOS workflow (`ExperimentWorkflow`): the DBOS workflow slices by sample (enumerating `dataset_samples` via streaming keyset pagination, not loading them all at once), each LLM call enters the Redis pending admission backlog, and the worker-side dispatcher admits it into the BullMQ `llm` ready queue only when a concurrency slot is available; model rate limiting goes through Redis; result aggregation is completed within a DBOS step (see [03 ┬¦3.1](03-orchestration.md#31-experimentworkflow)). When `launcher.launch` throws, the service's catch changes the status directly to `failed`, so from the user's view there is no intermediate state where "it is pending the instant it is created and only advances afterward".

Before the experiment is successfully written, the system first freezes the target prompt version (`prompt_versions.is_frozen=true`). Freezing only affects the prompt version's execution contract and no longer advances the Prompt Version to the `testing` status; the experiment's own run status is expressed by `ph_runs.experiments.status`.

## 4. The run process

- The DBOS workflow splits samples into stable LLM jobs and writes them into the pending admission backlog. The dispatcher admits pending jobs to the BullMQ `llm` ready queue according to keyed concurrency leases, and the worker pulls only ready jobs. Immediately before the provider request, admitted jobs use their admission lease as the concurrency reservation and call the Redis limiter in pre-reserved mode for final RPM / TPM accounting/checks; non-admitted direct calls still reserve concurrency in the limiter. The experiment-level RPM / TPM / concurrency are applied after taking the min with the model level; the model-level quota is the ceiling, and all call channels share the same Redis limiter key produced by `LimiterKeyStrategy` for that model/project context (OSS default: `model:<modelId>`)
- Multiple experiments can run concurrently at the same time, and the platform ensures the total across all experiments never exceeds the model's capacity
- When a single sample's LLM call fails, it first does exponential-backoff retries within `invokeLLM` according to the experiment-configured retry count (only effective for 408 / 429 / 5xx / network errors); after exhausting the internal retries it is handed off to BullMQ job-level attempts. `RateLimitExceededError` does not consume internal retry counts and is passed through as-is to BullMQ for re-queuing. Once BullMQ reports a job as terminal, the corresponding sample must be materialized as a terminal `run_results` row; if the worker finalization event or run-result write was missed, `ExperimentWorkflow` reconciles the terminal job payload into an idempotent `status='failed'` row instead of waiting forever
- The user can **stop** an experiment mid-run (writing `stop` to `ph_runs.experiments.control_state`, which the DBOS workflow senses at a step boundary and exits); after stopping it can be **resumed** (writing `resume` to the control state and starting a new resume workflow instance, which picks up from unfinished samples through stable `runResultId` idempotency). When stop is observed while a batch has already been enqueued, the workflow removes not-yet-started jobs for that batch from both the pending admission backlog and the BullMQ `llm` ready queue before finalizing `stopped`; jobs already active in a worker are allowed to finish and write terminal run results, while BullMQ-terminal jobs whose business row is missing are materialized as failed run results by the workflow. The stopped progress snapshot counts every completed or failed sample and leaves only the removed / not-yet-enqueued samples for resume. Legacy `cancel` control actions are accepted as a stop alias and also land in `stopped`.
- Run-progress refresh is driven by a fixed 5-second poll from the frontend. While running it issues `GET /experiments/:experimentId` + `GET /experiments/:experimentId/run-results` every 5s throughout the `running` period; once the run ends the control auto-hides. After each batch completes, the backend writes `experiments.processed_samples` / `failed_samples` / `metrics` as a snapshot; for experiments with `status='running'`, the detail and list endpoints can aggregate `ph_runs.run_results` by `(source='experiment', source_id=experimentId)` and use the real-time aggregation to override the DTO's `processedSamples / failedSamples / metrics`; terminal-state experiments read the `experiments` table fields directly. `failed_samples` must follow the run-result failure definition in [30 ┬¦1](30-run-results.md#1-page-role): `status='failed'`, `judgment_status='parse_error'`, or `judgment_status='judge_error'` when an expected result exists count as failed samples.

## 5. State machine

```
Created ŌåÆ running ŌåÆ success / failed / stopped
stopped ŌåÆ resume ŌåÆ running
```

There are only four legal statuses, `running / success / failed / stopped`, and **no `pending`**. The experiment lands `status='running' + started_at=NOW()` the moment it is written in `repository.createExperiment`; when the DBOS workflow fails to start, the service's catch changes the status to `failed`, so from the user's view there is no intermediate state where "it is pending the instant it is created and only advances afterward". `markStartedStep` continues to idempotently write `status='running' + started_at` inside ExperimentWorkflow as a backstop for the restart path.

## 6. Detail page

After entering an experiment you can see:

- **Progress and status**: completed count / total count, failed count, run duration
- **Metric cards**: Accuracy, Precision, Recall, F1
- **Multi-class metrics**: if the output schema declares a "decision field" and the field is an enum, the platform splits metrics by enum value (per-class breakdown) to see how each class performs individually
- **Results table**: one row per sample; the sample inputs are **expanded into N columns** in the order of the fields with role Ōłł `text | image | image_url | image_base64` in the dataset's `field_schema`, with column name = field name (e.g. `ocr_text` / `image_url`). The header reuses `ModalityIcon` to distinguish text / image types, and the image subtypes (`image / image_url / image_base64`) are noted in small text next to the column name; a text cell shows the truncated text (hover the tooltip to see the full content), image / image_url cells show a thumbnail + URL text, and an image_base64 cell shows a `(base64 ┬Ę ŌĆ”chars)` placeholder; the remaining fixed columns: model output, expected output, chain success state, judgment result (correct / incorrect only), failure reason (only for run-result failures, with a summary and a detail line), latency, creation time. Click a row to expand the run-result detail sidebar and see the full content
- **Filtering and sorting**: filter by judgment status, error type, latency, token usage
- **Export**: the detail page offers two download-package options, CSV and JSONL. Both options return a ZIP package that always includes the experiment summary as `summary.csv`; the selected format controls the run-results detail file (`run-results.csv` or `run-results.jsonl`). The detail export streams every row matching the current experiment result filters, is not limited to the visible page, and rehydrates offloaded run-result payload fields through the shared reader seam. The older single-file summary export and single-file run-results export endpoints remain available for automation clients.
- **Quick actions**:
  - Start a **canary release** based on the current experiment (carrying in the prompt version, used with event type `from_canary` when it proceeds to a production release)
  - **Directly create a production release** based on the current experiment (event type `from_experiment`, skipping the canary)

## 7. Judgment details

The basis on which the platform judges whether a single sample is "correct":

- The **output schema** defined by the prompt version ŌåÆ determines which field to extract from the LLM return as the "decision value (Decision)"
- The **judgment rules** defined by the prompt version ŌåÆ compare the "decision value" against the "expected output" in the sample

Judgment modes include: exact match, contains, value mapping, threshold comparison, custom expression, and equality. When judgment fails, the model's raw output is still retained for manual analysis.

**Judgment execution location**: the judgment logic runs on the **worker side**, in the same process and same transaction as the LLM call. In the `enqueueBatch` step the server packs `output_schema + judgment_rules + expected_output` into `LlmJobPayload.judgment` and passes it through to the worker; `expected_output` is read from the prompt rules' canonical `expectedField` when present, otherwise from the dataset field whose role is `expected_output`, and only falls back to the legacy literal key `expected_output` when neither source declares a field. Historical prompt JSON using `expected_field` or first-rule `value` remains readable, but new writes use `expectedField`. Once the worker obtains `parsed` in `invokeLLM` it immediately calls `evaluateJudgment` from `@proofhound/judgment` and writes `decision_output / is_correct / judgment_status` together with `expected_output` into `ph_runs.run_results` (a single INSERT, with the three pieces landing in the DB in sync). **Rationale**: `ph_runs.run_results` is append-only and cannot be UPDATE-d after the fact; letting the server run judgment within the aggregation step would require breaking the invariant or adding a shadow table.

**Stability of `runResultId`**: `runResultId = uuidv5(`${experimentId}:${sampleId}`, NS)`, staying constant across retry / resume / worker re-dispatch. The pending backlog dedupe key and BullMQ `jobId` both use `runResultId`, so the same sample is not enqueued twice across pending/ready transitions; on INSERT the worker uses `INSERT ... SELECT ... WHERE NOT EXISTS (... id = $1)` as a backstop to avoid multiple rows with the same id in run_results (because run_results is partitioned by month, a single-column id cannot have a UNIQUE constraint).

**Control signals**: at the Service layer, `stop` / `resume` only write `ph_runs.experiments.control_state` and do not directly modify the terminal `status`; the source of truth for the `status` state machine is advanced to a terminal state (`stopped` / `success` / `failed`) by `ExperimentWorkflow` after it senses them at a step boundary. Legacy `cancel` actions are normalized to `stop`, so cancellation is recoverable through the same `stopped -> resume -> running` path. When stop is sensed during a polling batch, the workflow removes not-yet-started BullMQ jobs for that batch and waits only for already-started jobs to produce terminal run results before taking the final stopped snapshot. During every polling batch, if a BullMQ job has already reached `completed` / `failed` but the corresponding business `run_results` row is missing, the workflow records an idempotent failed run result from the queued payload before rechecking progress; the stop cleanup additionally removes that stale terminal job. This guarantees that the "user operation" and the "actual run status" do not become inconsistent, while preventing queued-but-unstarted samples from continuing after stop and preventing lost worker finalization events from leaving the UI in `running` until timeout. For `resume` / `retry`, the Service conveniently calls `ExperimentLauncher.resume / retry` to start a new workflow instance.

## 8. Relationship with other menus

- **Prompt / dataset / model**: an experiment is a combined product of these three
- **Optimization**: each optimization round embeds an experiment, reusing the same engine
- **Canary release**: an experiment is a common prerequisite for a canary; many teams agree that "if experiment metrics are not met, starting a canary is not allowed"
- **Production release**: releasing directly from an experiment (`from_experiment`) is a fast path, suitable for the scenario where "the experiment set is already very representative of the production distribution"
- **Run results**: each result an experiment produces **simultaneously** writes a run result, facilitating unified online / offline metric aggregation
- **Run results**: the metrics and call details an experiment produces can all be traced in run results
