# 26 · Connectors

## 1. Page Role

Connectors describe the data path between ProofHound and external systems. A release line consumes requests through an upstream connector and pushes results through downstream connectors; the canary candidate and the production lane can share the upstream connector within the same release line.

Connectors are instance-level resources, with their configuration stored directly in `ph_assets.connectors`.

## 2. Types

| Type    | Input side              | Output side             |
| ------- | ----------------------- | ----------------------- |
| Redis   | Read List / Stream      | Write List / Stream     |
| Kafka   | Subscribe to a topic    | Publish to a topic      |
| Webhook | Expose an HTTP POST entry | Call downstream HTTP  |

The input and output direction cannot be modified after creation.

## 3. Configuration and Credentials

- Non-sensitive configuration is stored in `config`.
- Sensitive credentials are stored in `config_encrypted`.
- API responses and frontend detail views do not echo back sensitive credentials.
- When editing, leaving a sensitive field empty means the existing ciphertext is retained; only filling in a new value updates the ciphertext.

Redis / Kafka connectors store instance-level configuration such as host, port, bootstrap servers, protocol, and username directly.

## 4. List and Operations

The list displays:

- Name
- Direction
- Type
- Health status
- Last probe time / error
- Last updated time

Available operations:

- Create.
- Edit.
- Delete, rejected when occupied by an active release line, a running production, or a running canary candidate.
- Bulk delete.
- Peek for input connectors.
- Connectivity probe.

## 5. Webhook Input Connectors

Webhook input connectors are hosted by a standalone `apps/webhook` service. The management API is only responsible for configuration; public requests enter the webhook app.

Modes:

- Synchronous: after the upstream POSTs, it waits for the LLM call to complete and returns the result.
- Asynchronous: after the upstream POSTs, it immediately returns a `call_id`, and writes a short-lived receipt once background inference finishes; the caller queries with the `call_id`.

Path:

```text
/webhooks/:webhookPath
```

`webhook_path` is generated by the system and guaranteed to be unique within the current connector. The page displays the relative path, and call examples use the `$PROOFHOUND_API_ORIGIN` placeholder.

### 5.1 Authentication and Token Model

The webhook entry uses a **per-connector webhook token**, which is **fully independent from and not reusable with** the user token (shared by HTTP API / MCP) managed in [34 Settings](34-settings.md):

- Physically stored in `ph_core.tokens with scope='webhook' AND connector_id=<this connector id>`, validated by `ConnectorContextResolver` ([08 §3.4](08-saas-adapter-boundary.md#34-connectorcontextresolver))
- The credential lifecycle is **owned by the connector resource**: generated when the connector is created, and cascade-cleaned when the connector is deleted; it is not managed on the 34 Settings page
- **A single connector supports multiple valid tokens coexisting in steady state**—used for per-consumer distribution: one webhook entry can issue different tokens to multiple consumers, and perform call statistics and auditing on a per-token basis
- The IP allowlist is stored as connector configuration and evaluated by the webhook ingress (in parallel with token authentication)
- The validation failure code is `invalid_webhook_token` (distinct from the user token failure code `invalid_api_token`)

### 5.2 Token Management

The connector detail page embeds a "Webhook Tokens" section, which carries all lifecycle operations for webhook tokens.

**Automatic generation of the first token**

- When a webhook input connector is created, the system automatically generates 1 webhook token as the first credential
- The creation response reveals the plaintext token one time, and the user must copy and save it immediately; afterward the UI no longer displays the full plaintext (only the `prefix`, e.g. `ph_wh_xxxxxxxx`)

**Adding a token**

- The "Add token" button on the detail page: the user fills in `name` (required, max length 64) and an optional `expires_at` (e.g. "this token expires in 90 days")
- The generation response reveals the plaintext one time, and the user copies and saves it; afterward only the prefix is displayed
- Token `name` is not forced to be unique within a single connector, but users are advised to choose a semantic identifier (e.g. `stripe-prod` / `internal-monitor`) for easier management

**Revoking a token**

- Each row in the token list on the detail page has a "Revoke" button
- Revocation takes effect immediately (`revoked_at = now()`), and subsequent webhook requests with that token return 401 `invalid_webhook_token`
- The revoked token row is **retained** for audit traceability (not physically deleted); the FK reference from `ph_runs.run_results.webhook_token_id` remains valid

**Rotation**

- There is no built-in "canary transition / grace period" mechanism; rotation = the user performs "add token → switch upstream consumers over → revoke the old token" themselves
- The timing is controlled by the user; `expires_at` is a security upper bound that the user proactively sets, not a rotation transition window

**List display fields**

Each token row displays: `name` / `prefix` / `created_at` / `last_used_at` / `expires_at` (nullable) / call count (links to §5.3) / Revoke button.

### 5.3 Per-Token Call Statistics

The `webhook_token_id` column of the `ph_runs.run_results` table records "which webhook token triggered this call" (see [06 §5.3](06-database-schema.md#53-ph_runsrun_results) and [08 §3.4](08-saas-adapter-boundary.md#34-connectorcontextresolver)):

- The token list on the connector detail page aggregates and displays by `webhook_token_id`: call count, last call time, success rate
- The monitoring page `/monitoring` can produce finer-grained charts by `webhook_token_id` (call volume trend, error distribution)
- HTTP API / MCP entries write NULL; these statistics do not cover those two entry types
- When a webhook token is deleted, the run_results rows are **retained**, and `webhook_token_id` is automatically set to NULL (audit integrity)

## 6. Peek

Peek is used to inspect the structure of the most recent messages on an input connector without consuming them.

- Redis List: `LRANGE key 0 N-1`.
- Redis Stream: `XREVRANGE key + - COUNT N`.
- Kafka: a temporary consumer group seeks to the latest N messages without committing the offset.
- Webhook: Peek is not provided, because it is not a persistent queue.

After a successful Peek:

- `health_status='healthy'`
- Write `last_probed_at`
- Clear `last_probe_error`
- Save `lastPeekMessage` / `lastPeekPayloadSchema` / `lastPeekedAt` / `lastPeekMessageCount` in `config`

After a failed Peek:

- `health_status='unhealthy'`
- Write `last_probed_at` / `last_probe_error`
- Clear the latest message snapshot

The response must not contain connection credentials. The application log only records the count, source, and an error summary, not the message body.

## 7. Relationship with Releases

- A release line must select one upstream connector; the upstream connector cannot be changed after the release line is created.
- The same upstream connector belongs to at most one active release line at the same time.
- Within the same release line, one production lane and one canary candidate lane are allowed to share the upstream connector.
- Multiple downstream connectors can be selected; a canary candidate can inherit the production downstream, switch to a test downstream, or add an observation downstream.
- A queue connector release must configure the request external ID field, field mapping, filter rules, and traffic mode.
- A webhook connector enters production directly on its first release; subsequently, adding a canary on a production line also uses the release routing configuration.
- Filter rules, variable mapping, and output mapping are completed in the release configuration.

## 8. Health Probe

The health probe is a lightweight read-only verification:

- Redis: connection and key type.
- Kafka: topic metadata.
- Webhook output: target URL reachability can be verified on demand.

Periodic probing only updates the connector's own health fields, for display on the list and detail pages.
