# 33 · Quick start

## 1. Page role

Quick start is a "zero to one" entry point: the user uploads a dataset with expected outputs, fills in a task description, and selects an experiment model and an analysis model. The system automatically creates the dataset, the prompt container, and the optimization task.

Quick start uses the default local project.

## 2. Entry points

The entry point appears in the fixed top action bar, on the Local admin app home page, or in resource empty states. On completion it navigates to the optimization detail page by default.

## 3. Required inputs

### 3.1 Dataset

- Dataset file: CSV / TSV / JSONL / JSON array.
- Field roles: text, image, image URL, image base64, expected output, metadata.
- There must be exactly one field marked as the expected output.

### 3.2 Task description

A short one-line description that tells the analysis LLM what task this dataset is meant to accomplish. For example:

- `Classify customer service conversations into 5 intent labels`
- `Classify support tickets into five intent labels`

### 3.3 Prompt language

Values:

- `zh-CN`
- `en-US`

It controls the language of the LLM-facing instructions that the system auto-generates. It does not translate the data, field names, variable names, JSON keys, or task description that the user uploaded.

### 3.4 Models

The two models are selected independently:

- Experiment model: the model actually invoked in each experiment round.
- Analysis model: the model used for optimization analysis and prompt generation.

The two can be the same. When choosing a quick preset, you must fill in the credentials and quota, and a connectivity test before saving is supported.

### 3.5 Optimization parameters

- Optimization goals: at least one.
- Maximum rounds: default 3.
- Strategy: fixed to `error_pattern_analysis`.
- Advanced run parameters: RPM / TPM / concurrency / temperature / per-sample timeout.

## 4. Automation after submission

The backend runs in sequence:

1. Persist the dataset.
2. Call the analysis model to generate names for the prompt and the optimization task; fall back to a template on failure.
3. Create the prompt container.
4. Create the optimization task, with `from_dataset_only` as the starting point.
5. Start the optimization DBOS workflow.
6. Return `{ projectId, datasetId, promptId, optimizationId }`.

The first prompt version is generated by the first round of optimization; quick start itself does not define separate first-version generation logic.

## 5. Automatic naming

Naming LLM input:

- Task description.
- The first N samples of the dataset.
- `promptLanguage`.

Output:

- `promptName`
- `optimizationName`

Failure fallback:

- Chinese: `{dataset_filename_stem} 初版提示词` / `{dataset_filename_stem} 优化 #1`
- English: `{dataset_filename_stem} Initial Prompt` / `{dataset_filename_stem} Optimization #1`

The naming LLM call must comply with the LLM call logging contract in [05 Logging specification](05-logging.md).

## 6. Failure handling

| Failure location | Behavior |
| -------- | ---- |
| Dataset persistence | Fails as a whole, the user is allowed to retry |
| Naming LLM | Non-blocking, uses the fallback template |
| Prompt / optimization creation | Keeps the already-created dataset, prompts the user to continue through the regular menus |
| Workflow startup | Keeps the optimization task, marks its status as failed |

Every failure must write a structured application log that contains at least the step number and the error type.

## 8. Invariants

- A single quick start produces exactly one dataset, one prompt, and one optimization task.
- All required fields are mandatory.
- Automatic naming failure must fall back and must not cause overall failure.
- The resources produced are fully equivalent to resources created through the regular path, with no special `is_quick_start` marker.
