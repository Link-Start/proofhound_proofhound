## 1. General conventions

| Dimension   | Choice                                                                                                                                                                  |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Primary key | `UUID`, generated by `gen_random_uuid()`                                                                                                                                |
| Time        | `TIMESTAMPTZ` + default `NOW()`; written in UTC, formatted for display on the frontend                                                                                  |
| String      | `TEXT`; length constraints validated by the application-layer DTO                                                                                                       |
| Enum        | `TEXT` + `CHECK`, not native ENUM                                                                                                                                       |
| Deletion    | Prompt / dataset use explicit `active` / `archived` existence states; permanent deletion is a separate physical-delete flow guarded by a deletion hook and confirmation |
| JSON        | `JSONB`; dynamic config, field mappings, and snapshots all go through JSONB                                                                                             |
| Encryption  | Application-layer encrypted fields are named with the `_encrypted` suffix                                                                                               |
| Naming      | schema / table / column are all snake_case                                                                                                                              |
| Extensions  | Relies only on standard PostgreSQL capabilities; UUID uses `pgcrypto`                                                                                                   |

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;
```

Deletion and archival policy:

| Resource     | Archive behavior                                                                                                                                                                                                                                                      | Permanent delete behavior                                                                                                                                                                                                                                                                                      | Required pre-delete hook                                                                                                                                                                                                                                                                                                                                                |
| ------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Prompt shell | `active -> archived` removes the prompt from new experiment, optimization, and release creation. Existing experiments, optimizations, release lines, and run results continue from their prompt version / release snapshots.                                          | Deletes the prompt shell and versions, deletes affected experiments / optimizations and their owned descendants, and stops any running production or canary release lanes that depend on the prompt. Stopped releases keep their event snapshots for history unless the release itself is permanently deleted. | The deletion hook returns an impact plan. OSS must list affected release lines first, then experiments and optimizations before confirmation; release impact is listed only at release-line granularity, not as individual canary / production events. SaaS may extend the same hook with releasable storage estimates without adding OSS-only billing or plan modules. |
| Dataset      | `active -> archived` removes the dataset from new experiment / optimization creation and from release creation paths that depend on that dataset through a prompt binding. Existing experiments and optimizations keep running from the data they already referenced. | Deletes the dataset and samples, clears prompt default-dataset bindings where needed, and deletes affected experiments / optimizations and their owned descendants. Releases are not dataset children and are not directly stopped by dataset deletion.                                                        | The deletion hook returns an impact plan. OSS must list affected experiments, optimizations, prompt bindings, and sample counts before confirmation. SaaS may extend the same hook with releasable storage estimates.                                                                                                                                                   |
| Experiment   | No archive state; use stop for runtime control. Legacy cancel actions are accepted as a stop alias and land in `stopped`.                                                                                                                                                          | Deletes the experiment as a child work item and deletes its owned run results / annotations. Release attribution snapshots continue to display from release event snapshots.                                                                                                                                   | The hook may return a small owned-descendant count for confirmation, but does not block on prompt / dataset / release references.                                                                                                                                                                                                                                       |
| Optimization | No archive state; use stop / cancel for runtime control.                                                                                                                                                                                                              | Deletes the optimization as a child work item, including optimization round steps, optimization LLM run results, and generated sub-experiments owned by the optimization.                                                                                                                                      | The hook may return owned-descendant counts for confirmation, but does not block on prompt / dataset references.                                                                                                                                                                                                                                                        |
| Release line | `archive_line` is the non-destructive "hide from active operation" state after lanes are no longer running; `unarchive_line` restores the line to `stopped`, never directly to `running`.                                                                             | Deletes the release line aggregate, variants, events, release run results, and annotation tasks owned by that release. This is a direct release-management action exposed from the release detail settings tab danger area; prompt permanent delete only stops running lanes.                                  | The user must type the release name before confirmation. The deletion hook may provide an impact plan for backend-side delete preparation and future extended presentations.                                                                                                                                                                                            |

The deletion hook is a thin extension point used by OSS first: it must be invoked for preview and again immediately before execution, using the same resource plan or a plan version/hash so the user cannot confirm a stale impact list. Permanent deletion is the only supported path that removes retained run results; otherwise run results are immutable once written and manual annotations stay in `ph_runs.annotations`.

## 2. Schema partitioning

```sql
CREATE SCHEMA ph_core;      -- Default local project / user token / webhook token
CREATE SCHEMA ph_assets;    -- Model / dataset / prompt / connector
CREATE SCHEMA ph_runs;      -- Experiment / optimization / run result / annotation
CREATE SCHEMA ph_releases;  -- Release / annotation task
```

The business schemas are fixed to the four above; control-plane schemas are not expanded in the OSS database SPEC.

## 3. ph_core â€” Local project / Token

### 3.1 `ph_core.projects`

`projects` holds only the default local project and is the business data boundary. Resources within the project are associated via `project_id`; a future external control plane can integrate through this ID without rewriting the business Service.

```sql
CREATE TABLE ph_core.projects (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name        TEXT NOT NULL,
  description TEXT,
  type        TEXT NOT NULL DEFAULT 'classification'
              CHECK (type IN ('classification', 'generative', 'agent')),
  status      TEXT NOT NULL DEFAULT 'active'
              CHECK (status IN ('active', 'archived')),
  created_by  UUID NOT NULL,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  archived_at TIMESTAMPTZ,
  deleted_at  TIMESTAMPTZ
);

CREATE INDEX idx_projects_active
  ON ph_core.projects (status)
  WHERE deleted_at IS NULL;
```

### 3.2 `ph_core.tokens`

`tokens` carries two kinds of credentials at once: the local admin app user token, and per-connector webhook tokens. The two are distinguished by `scope` and map to different entry resolvers (see [08 Â§3](08-saas-adapter-boundary.md#3-extension-point-list)).

- `scope='user'`: local admin app user credential, **a single token can be used at both the HTTP API and MCP entries simultaneously**; in OSS `project_id` is always NULL (single workspace, no business meaning) and `connector_id` must be empty. The SaaS form may attach `project_id` for a per-token-per-project association, but OSS does not write it.
- `scope='webhook'`: per-connector inbound credential, must be bound to `project_id` and `connector_id`; **the same connector supports multiple valid tokens coexisting in steady state**, used for per-consumer distribution (one webhook may issue different tokens to multiple consumers, with call statistics and auditing tracked per token).

```sql
CREATE TABLE ph_core.tokens (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  scope           TEXT NOT NULL CHECK (scope IN ('user', 'webhook')),
  project_id      UUID REFERENCES ph_core.projects(id) ON DELETE CASCADE,
  connector_id    UUID REFERENCES ph_assets.connectors(id) ON DELETE CASCADE,
  name            TEXT NOT NULL CHECK (char_length(name) <= 64),
  token_hash      TEXT NOT NULL UNIQUE,
  token_encrypted TEXT,
  prefix          TEXT NOT NULL,
  ip_whitelist    JSONB,
  last_used_at    TIMESTAMPTZ,
  expires_at      TIMESTAMPTZ,
  created_by      UUID NOT NULL,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  revoked_at      TIMESTAMPTZ,
  CHECK (
    (scope = 'user'    AND connector_id IS NULL) OR
    (scope = 'webhook' AND project_id IS NOT NULL AND connector_id IS NOT NULL)
  )
);

CREATE INDEX idx_tokens_project
  ON ph_core.tokens (project_id)
  WHERE revoked_at IS NULL;

CREATE INDEX idx_tokens_active
  ON ph_core.tokens (scope)
  WHERE revoked_at IS NULL;

CREATE INDEX idx_tokens_connector
  ON ph_core.tokens (connector_id)
  WHERE scope = 'webhook' AND revoked_at IS NULL;
```

The token plaintext is returned only at creation; `token_encrypted` is used to display a recoverable token on demand in the local admin app. Authentication comparison uses only `token_hash`.

Key invariants:

- User tokens are unlimited in count and can be used at both the HTTP API and MCP entries simultaneously; there is no longer a "at most one valid Global MCP token at a time" constraint.
- Webhook tokens do not reuse user tokens; coexisting in the same physical table is only to simplify auditing / hash uniqueness.

Lifecycle ownership:

- CRUD of `scope='user'` rows is owned by `TokenService` ([08 Â§3.5](08-saas-adapter-boundary.md#35-tokenservice)), overridden by `RemoteTokenService` in the SaaS form.
- CRUD of `scope='webhook'` rows is owned by connector resource management ([26 Connectors](26-connectors.md)), created / rotated / deleted along with the connector; `TokenService` does not read or write such rows.
- Cross-scope unified constraints (such as `token_hash UNIQUE` and `prefix` display rules) are guaranteed by table-level constraints, independent of lifecycle ownership.

## 4. ph_assets â€” Model / Dataset / Prompt / Connector

### 4.1 `ph_assets.models`

Model configuration belongs to the default local project.

```sql
CREATE TABLE ph_assets.models (
  id                               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id                       UUID NOT NULL REFERENCES ph_core.projects(id) ON DELETE CASCADE,
  name                             TEXT NOT NULL,
  provider_type                    TEXT NOT NULL,
  provider_model_id                TEXT NOT NULL,
  endpoint                         TEXT NOT NULL,
  api_key_encrypted                TEXT NOT NULL,
  context_window_tokens            INTEGER,
  rpm_limit                        INTEGER NOT NULL DEFAULT 60,
  tpm_limit                        INTEGER NOT NULL DEFAULT 100000,
  concurrency_limit                INTEGER NOT NULL DEFAULT 20,
  auto_concurrency                 BOOLEAN NOT NULL DEFAULT TRUE,
  input_token_price_per_million    NUMERIC(12, 6) NOT NULL DEFAULT 0,
  output_token_price_per_million   NUMERIC(12, 6) NOT NULL DEFAULT 0,
  capabilities                     JSONB NOT NULL DEFAULT '{}'::jsonb,
  extra_body                       JSONB NOT NULL DEFAULT '{}'::jsonb,
  is_active                        BOOLEAN NOT NULL DEFAULT TRUE,
  last_probed_at                   TIMESTAMPTZ,
  last_probe_error                 TEXT,
  created_by                       UUID NOT NULL,
  created_at                       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at                       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  deleted_at                       TIMESTAMPTZ,
  CHECK (context_window_tokens IS NULL OR context_window_tokens > 0),
  CHECK (rpm_limit = -1 OR rpm_limit > 0),
  CHECK (tpm_limit = -1 OR tpm_limit > 0),
  CHECK (concurrency_limit BETWEEN 1 AND 999),
  CHECK (input_token_price_per_million >= 0),
  CHECK (output_token_price_per_million >= 0),
  CHECK (jsonb_typeof(extra_body) = 'object')
);

CREATE UNIQUE INDEX idx_models_project_name_active
  ON ph_assets.models (project_id, name)
  WHERE deleted_at IS NULL;
```

`project_id + name` guarantees that the name of a non-deleted model is unique within the project.

When `auto_concurrency=TRUE` (the default), `concurrency_limit` is the concurrency ceiling, and the system self-tunes the actual effective concurrency within `[1, concurrency_limit]`; when `FALSE`, `concurrency_limit` is the hard concurrency ceiling. See [21 Â§6.1](21-models.md#61-auto-concurrency) for the semantics.

### 4.3 `ph_assets.datasets` / `dataset_samples`

```sql
CREATE TABLE ph_assets.datasets (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id     UUID NOT NULL REFERENCES ph_core.projects(id) ON DELETE CASCADE,
  name           TEXT NOT NULL,
  description    TEXT,
  status         TEXT NOT NULL DEFAULT 'active'
                 CHECK (status IN ('active', 'archived')),
  sample_count   INTEGER NOT NULL DEFAULT 0,
  field_schema   JSONB NOT NULL DEFAULT '[]'::jsonb,
  has_images     BOOLEAN NOT NULL DEFAULT FALSE,
  storage_prefix TEXT,
  created_by     UUID NOT NULL,
  created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  archived_at    TIMESTAMPTZ,
  deleted_at     TIMESTAMPTZ
);

CREATE UNIQUE INDEX idx_datasets_project_name_active
  ON ph_assets.datasets (project_id, name)
  WHERE deleted_at IS NULL;

CREATE TABLE ph_assets.dataset_samples (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  dataset_id  UUID NOT NULL REFERENCES ph_assets.datasets(id) ON DELETE CASCADE,
  data        JSONB NOT NULL,
  external_id TEXT,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

`dataset_samples` is one of the few tables that allow physical deletion, to support re-upload and bulk clearing of samples.

`datasets.storage_prefix` is no longer written on the import path without object storage; the column is retained only for backward compatibility, and new datasets do not depend on it (see [04 Â§4](04-postgresql.md#4-storage)). `datasets.status` is the formal dataset existence state (`active` / `archived`) and is independent from import progress. During import, samples only land in the staging table below, and the formal tables always contain only complete datasets.

#### 4.3.1 `ph_assets.dataset_imports` / `dataset_import_samples`

These support raw-upload asynchronous import and the legacy client-streamed fallback (see [22 Â§3.1.2](22-datasets.md#312-raw-upload--asynchronous-backend-import) and [22 Â§3.1.3](22-datasets.md#313-legacy-client-streamed-batched-import)): `dataset_imports` is the import session, and `dataset_import_samples` is the temporary staging before all rows are collected. On successful promotion, the staged samples are atomically promoted to `dataset_samples` and a `datasets` row is created within a single transaction.

```sql
CREATE TABLE ph_assets.dataset_imports (
  id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id          UUID NOT NULL REFERENCES ph_core.projects(id) ON DELETE CASCADE,
  dataset_id          UUID REFERENCES ph_assets.datasets(id) ON DELETE CASCADE,
  name                TEXT NOT NULL,
  description         TEXT,
  field_mappings      JSONB NOT NULL DEFAULT '[]'::jsonb,
  file_name           TEXT NOT NULL,
  file_size_bytes     BIGINT NOT NULL,
  content_type        TEXT,
  source_format       TEXT NOT NULL CHECK (source_format IN ('jsonl', 'csv', 'tsv', 'json', 'zip')),
  import_mode         TEXT NOT NULL DEFAULT 'batch'
                      CHECK (import_mode IN ('batch', 'raw_object')),
  raw_upload_session_id TEXT,
  raw_upload_expires_at TIMESTAMPTZ,
  raw_object_ref      JSONB,
  raw_upload_completed_at TIMESTAMPTZ,
  progress            JSONB NOT NULL DEFAULT '{}'::jsonb,
  job_id              TEXT,
  error_code          TEXT,
  error_message       TEXT,
  queued_at           TIMESTAMPTZ,
  started_at          TIMESTAMPTZ,
  completed_at        TIMESTAMPTZ,
  failed_at           TIMESTAMPTZ,
  aborted_at          TIMESTAMPTZ,
  declared_total_rows INTEGER,
  received_rows       INTEGER NOT NULL DEFAULT 0,
  status              TEXT NOT NULL DEFAULT 'created'
                      CHECK (status IN (
                        'created', 'uploading', 'uploaded', 'queued',
                        'parsing', 'importing', 'completed', 'failed', 'aborted'
                      )),
  created_by          UUID NOT NULL,
  created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_dataset_imports_project_status
  ON ph_assets.dataset_imports (project_id, status);

CREATE INDEX idx_dataset_imports_job_id
  ON ph_assets.dataset_imports (job_id)
  WHERE job_id IS NOT NULL;

-- sweep scans pre-worker sessions that have had no heartbeat for a long time
CREATE INDEX idx_dataset_imports_stale
  ON ph_assets.dataset_imports (status, updated_at)
  WHERE status IN ('created', 'uploading', 'uploaded');

CREATE TABLE ph_assets.dataset_import_samples (
  import_id   UUID NOT NULL REFERENCES ph_assets.dataset_imports(id) ON DELETE CASCADE,
  row_index   INTEGER NOT NULL,
  data        JSONB NOT NULL,
  external_id TEXT,
  PRIMARY KEY (import_id, row_index)
);
```

Fields and lifecycle:

- `import_mode='batch'` is the legacy client-streamed JSONL / CSV / TSV path where the frontend pushes parsed samples to `POST /dataset-imports/:id/batch`.
- `import_mode='raw_object'` is the browser-direct raw object path. `raw_upload_session_id` stores the provider-issued pending upload session id until upload-complete/abort/sweep; `raw_upload_expires_at` mirrors the provider expiry for cleanup; `raw_object_ref` stores the finalized object reference after `completeUpload` and before the worker finishes parsing/promoting it. The raw object is temporary and may be deleted after successful promotion.
- `status` follows the state machine in [22 Â§3.1](22-datasets.md#31-uploading-data). `failed` / `aborted` rows are retained for readable status and error reporting, but staging rows and temporary raw upload resources are removed best-effort.
- `progress` is a user-facing progress snapshot for long server-side work after byte transfer: it stores visible `phase`, `totalShards`, `completedShards`, and `committedRows` counters so the upload page can poll the session during promotion. It is informational for rendering, but `phase='finalizing' | 'offloading' | 'committing'` also tells abort cleanup to request cancellation instead of deleting staging under an active promotion transaction; `status` remains the authoritative lifecycle state.
- `dataset_id` is NULL during import and is backfilled to point to the newly created dataset after a successful `complete` promotion.
- `received_rows` is the count of staged parsed rows and also serves as progress; `updated_at` is the heartbeat advanced on batch writes and state transitions. `job_id` links a raw import to its BullMQ job for idempotent queueing/debugging. Lifecycle timestamps support status display without scanning worker logs.
- `field_mappings` is submitted by the frontend field wizard when the session is created, and `complete` finalizes `datasets.field_schema` based on it.
- `dataset_import_samples` is temporary staging: the primary key `(import_id, row_index)` provides idempotency for single-batch resends; it hangs under the session via `ON DELETE CASCADE`, so deleting the session clears the staged samples. After successful promotion, failure, or abort, staged samples are explicitly cleared.
- Stale import cleanup must also abort the pending upload session and/or delete the finalized raw object for `raw_object` sessions, best-effort. Provider-level `sweepPendingUploads(olderThan)` catches pending objects whose database session was never created or was lost before abort could run.

### 4.4 `ph_assets.prompts` / `prompt_versions`

```sql
CREATE TABLE ph_assets.prompts (
  id                        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id                UUID NOT NULL REFERENCES ph_core.projects(id) ON DELETE CASCADE,
  name                      TEXT NOT NULL,
  status                    TEXT NOT NULL DEFAULT 'active'
                            CHECK (status IN ('active', 'archived')),
  current_online_version_id UUID,
  default_dataset_id        UUID REFERENCES ph_assets.datasets(id) ON DELETE RESTRICT,
  created_by                UUID NOT NULL,
  created_at                TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at                TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  archived_at               TIMESTAMPTZ,
  deleted_at                TIMESTAMPTZ
);

CREATE UNIQUE INDEX idx_prompts_project_name_active
  ON ph_assets.prompts (project_id, name)
  WHERE deleted_at IS NULL;

CREATE TABLE ph_assets.prompt_versions (
  id                             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  prompt_id                      UUID NOT NULL REFERENCES ph_assets.prompts(id) ON DELETE CASCADE,
  version_number                 INTEGER NOT NULL,
  body                           TEXT,
  variables                      JSONB NOT NULL DEFAULT '[]'::jsonb,
  output_schema                  JSONB,
  judgment_rules                 JSONB,
  prompt_language                TEXT NOT NULL DEFAULT 'zh-CN'
                                 CHECK (prompt_language IN ('zh-CN', 'en-US')),
  parent_version_id              UUID,
  generated_by_optimization_id UUID,
  change_reason                  TEXT,
  is_frozen                      BOOLEAN NOT NULL DEFAULT FALSE,
  created_by                     UUID NOT NULL,
  created_at                     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  frozen_at                      TIMESTAMPTZ,
  UNIQUE (prompt_id, version_number)
);
```

`project_id + name` guarantees that the name of a non-deleted prompt is unique within the project. `prompts.status` is the prompt shell existence state (`active` / `archived`) and is independent from prompt-version freeze state.

Prompt versions no longer maintain version states such as draft / in-experiment / in-canary / released. All versions default to `is_frozen=false`; once referenced by an experiment, optimization, canary, or production release, the `body` / `variables` / `output_schema` / `judgment_rules` / `prompt_language` execution contract is frozen.

```sql
CREATE TABLE ph_assets.prompt_version_labels (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  prompt_id   UUID NOT NULL REFERENCES ph_assets.prompts(id) ON DELETE CASCADE,
  version_id  UUID NOT NULL REFERENCES ph_assets.prompt_versions(id) ON DELETE CASCADE,
  label       TEXT NOT NULL,
  label_type  TEXT NOT NULL DEFAULT 'custom'
```

Run results are a fact table; business result fields are not updated after writing. Manual annotation, locking, and submission status are all written to `ph_runs.annotations`.

`run_result_ids` is the unpartitioned idempotency registry for `run_results`:

- Writers first reserve `run_result_ids.id` with `INSERT ... ON CONFLICT DO NOTHING`.
- Only the writer that successfully reserves the id inserts the immutable `run_results` fact row.
- `run_result_ids.created_at` is copied into `run_results.created_at`, giving the partitioned table a stable partition key for the reserved run result id.
- This registry is required because PostgreSQL cannot enforce `UNIQUE(id)` on a range-partitioned table unless the unique key includes the partition key.

`run_results` is physically partitioned by `created_at`:

- First version uses UTC monthly range partitions such as `run_results_2026_06` for `2026-06-01 00:00:00+00 <= created_at < 2026-07-01 00:00:00+00`.
- The partitioned table primary key is `(id, created_at)` because PostgreSQL unique constraints on partitioned tables must include the partition key; global id uniqueness is enforced by `run_result_ids.id`.
- A `run_results_default` partition catches old or future rows if an operational migration has not created the exact month partition yet. Normal maintenance should create future month partitions before traffic reaches them; if a month partition becomes too large to maintain, switch the operational cadence to weekly partitions.
- Partitioning improves historical query pruning, retention, archival, and `DROP` / `DETACH PARTITION` maintenance. It does not make online storage quota checks safe; write paths must not compute storage by scanning `run_results` or other detail tables.

Partition-aware indexes:


- `release_version_id`: the release version, representing a publishable lifecycle unit within a release line. Canary results keep the candidate release version; after promotion, production results use a new production release version linked back to the candidate.

`annotations` points to a run result and is associated with a manually created canary or production annotation task. ## 6. ph_releases â€” Release / Annotation task

### 6.1 `ph_releases.release_lines`

A release line belongs to the default local project and is the aggregate root for the `/releases` list and detail views. It represents a long-lived release name that is unique within the project, bound to one prompt and one upstream connector; the upstream connector cannot be changed after creation. The line itself stores only aggregate status and the current lane pointers; all executable configuration and operation history are written to `release_line_events`.

Key field semantics:

- `name` / `description`: the release name and description seen by the user.
- `prompt_id` / `prompt_name` / `prompt_snapshot`: the prompt the line belongs to and its snapshot at creation. If the prompt is archived, running releases continue unchanged. If the prompt is permanently deleted, running lanes that depend on it are stopped first and historical events continue to display from snapshots unless the release line itself is permanently deleted.
- `input_connector_id` / `input_connector_name` / `input_connector_type` / `input_connector_snapshot`: the line's upstream connector and snapshot. Changing the upstream requires creating a new line.
- `current_production_event_id`: the event currently shown in the production slot, nullable. It may reference a `running`, `stopped`, or `archived` production event; only `running` referenced events are executable by the runner / webhook runtime.
- `active_canary_event_id`: the event currently shown in the canary slot, nullable. It may reference a `running`, `stopped`, or `archived` canary event; both `running` and `stopped` canaries occupy the active canary slot for replacement / resume semantics, while `archived` canaries are retained only for detail-page visibility until unarchive.
- `status IN ('running', 'stopped', 'archived')`; `running` means at least one production or canary lane is currently running. Canary-only, production-only, and production-with-canary shapes are derived from `current_production_event_id`, `active_canary_event_id`, and the referenced event rows.

Key constraints:

- `project_id + name` guarantees release name uniqueness within the project.
- An upstream connector belongs to at most one non-`archived` release line at a time.
- A prompt has at most one release line with a running production lane at a time.
- A single release line has at most one active canary candidate at a time.

### 6.2 `ph_releases.release_versions`

`release_versions` is the source of truth for the user-facing release version. It solidifies one candidate or production publish lifecycle under a release line into a stable ID and structured version fields, used to query run results, create annotation tasks, and display quality metrics across canary and production stages.

Key field semantics:

- `release_line_id`: the release line it belongs to.
- `kind IN ('candidate', 'production')`: candidate release versions are canary attempts; production release versions are formal published versions.
- `production_version_number`: filled only for production release versions, displayed as `v1`, `v2`, etc. It is monotonically increasing within the release line.
- `target_production_version_number`: the production version this release version is trying to become or already is. Before the first production, candidates targeting `v1` display as `v0.1`, `v0.2`, etc.; after `v1`, candidates targeting `v2` display as `v1.1`, `v1.2`, etc.
- `candidate_number`: filled only for candidate release versions and unique within `release_line_id + target_production_version_number`.
- `promoted_from_release_version_id`: for production versions produced by canary promotion, points to the candidate release version that was promoted.
- `prompt_id` / `prompt_name` / `prompt_version_id` / `prompt_version_number` / `prompt_snapshot` / `prompt_version_snapshot`: the prompt version and snapshots the release version corresponds to.
- `model_id` / `model_snapshot`: the model and snapshot the release version corresponds to.

Key constraints:

- `production_version_number` is unique for production release versions within a single release line.
- `candidate_number` is unique for candidate release versions within the same `release_line_id + target_production_version_number`.
- The same prompt version and model may appear in multiple release versions; release versions are lifecycle records, not a deduplicated content-combination table.

### 6.3 `ph_releases.release_line_events`

`release_line_events` is the single source of truth for release operation history and executable snapshots. Canary and production are not statuses but `lane_type`; status only describes the current lifecycle of that lane / operation record.

Key field semantics:

- `release_line_id`: the release line it belongs to.
- `lane_type IN ('production', 'canary')`: the lane the event acts on.
- `operation`: the release operation type, e.g. create production, create canary, adjust traffic, pause, resume, cancel, promote, rollback, take offline, archive.
- `status IN ('running', 'stopped', 'completed', 'failed', 'cancelled', 'archived')`, no longer using `pending`.
- `terminal_reason`: terminal-state reasons such as `replaced` / `rolled_back` / `force_stopped` / `promoted` / `cancelled` / `archived` / `error`, nullable.
- `source_event_id`: records the source when derived from a historical event, e.g. canary promotion or rollback.
- `supersedes_event_id`: records the old event when a new event replaces an old running production.
- `rollback_target_event_id`: the rollback target event.
- `release_version_id`: the release version this event belongs to. Traffic ratio, mode, pause/resume, and runtime-only changes reuse the current release version; new canary candidates, direct productions, promotions, rollbacks, and version-affecting config changes create a new release version.
- `operation='restore_to_production' | 'restore_to_canary'`: copies a historical executable snapshot into the current production / canary slot and creates a new release version without mutating the historical event or its run results.
- `traffic_mode IN ('split', 'dual_run')` and `traffic_ratio`: the canary routing configuration; subsequent adjustments allow `0 <= traffic_ratio <= 1`.
- `prompt_id` / `prompt_name` / `prompt_version_id` / `prompt_version_number` / `prompt_snapshot` / `prompt_version_snapshot`: every event stores the prompt and version snapshots. A running release reads only the event snapshot and does not rely on still being able to join to the prompt asset tables.
- `model_id` / `model_snapshot`: the model snapshot.
- `input_connector_id` / `input_connector_snapshot`: the upstream connector snapshot.
- `output_connector_ids` / `output_connector_snapshots` / `output_mapping`: the downstream configuration snapshot.
- `variable_mapping` / `filter_rules` / `record_mode` / `record_categories` / `external_id_field` / `run_config`: the input routing and run configuration snapshot. `record_mode='all'` records all release run results; `record_mode='selected_categories'` records according to selected prompt-output classification values in `record_categories`, matched to `run_results.decision_output`. Online release traffic may produce `decision_output` without a true `expected_output`; in that case correctness remains unknown. Legacy `correct_only` values are accepted only for compatibility and must not be surfaced as the current product semantics, because online traffic usually does not know the correct label.
- `metrics` and `total_received` / `total_processed` / `total_filtered` / `total_correct` / `total_errors`: the runtime aggregation snapshot. Counts accumulate independently per `release_line_events.id`; a new event written by a config change, traffic change, promotion, or rollback does not inherit the old event's counts, and the old event keeps its own run results and counts for historical traceability.

Key constraints:

- A single release line has at most one `lane_type='production' AND status='running'` event at a time.
- A single release line has at most one `lane_type='canary' AND status IN ('running', 'stopped')` event at a time.
- A prompt has at most one running production lane at a time.
- `operation='rollback'` must have a `rollback_target_event_id`.
- `operation='promote_canary'` must have a `source_event_id`.
- Submitting a new production, canary promotion, config change, and rollback all write a new event, and update the superseded running production event to `stopped`.

### 6.4 Retirement of legacy release tables

The historical `ph_releases.canary_releases` and `ph_releases.production_release_events` are no longer used as runtime read/write or query sources. Current release business reads and writes use `release_lines` / `release_line_events` as the source of truth; if historical data needs to be retained, it should first be backfilled into the new tables in a migration, and run results and annotation tasks should be migrated to reference `release_line_events.id`. REST / MCP / Webhook / runner no longer read or write the legacy tables, and the legacy table data may be cleared after backfilling is complete.

### 6.5 `ph_releases.annotation_tasks`

Annotation tasks are created manually by the user at `/annotations/new`, and are not auto-created by canary releases, production releases, the queue runner, or the webhook runner. The new-task flow is: task name, release name and release version, then annotation data amount:

- Release line: `release_lines.id`, determined by the selected release name.
- Release version: `release_versions.id`, confirming the data to be annotated via the version label, prompt version, and model.
- Version scope: the new-task UI uses `exact` by default; `journey` remains a backend-compatible scope and expands a production version to compatible candidate versions targeting that production plus the production version itself.
- Data scope: the new-task UI uses `scope='all'` for the selected release version's current run results across canary and production lanes. Historical / compatibility tasks may still use `scope='canary'` for canary data or `scope='online'` for production data.
- Sample count: before creation, the system must return the maximum number of annotatable run results for the selected version and the run-result counts grouped by `run_results.decision_output`. At creation, selected run results are written to `ph_runs.annotations`.
- Sampling mode: `random` randomly samples the requested number from the selected version's matching run results; `per_category` lets the user specify requested counts per prompt classification category, bounded by the current `decision_output` count for each category.
- Annotation schema: always contains the `expected_output` field, whose field type is select; the optional classifications are derived from the release version's prompt version snapshot `output_schema.fields[].value`, preferring output fields with `isJudgment=true`. Journey scope is allowed only when every included release version has compatible category options.

Annotation tasks bind `release_version_id` first; for backward compatibility with historical data, they may also bind `release_line_event_id`, `canary_id`, or `production_release_event_id`. The status is `active` / `completed` / `archived`.

## 7. Key invariants

- A prompt has at most one running production lane at a time.
- An upstream connector belongs to at most one active release line at a time.
- Release names are unique within a project.
- Within a single release line, one running production lane and one running canary candidate lane are allowed to share the upstream connector.
- A single release line has at most one active canary candidate at a time; both `running` and `stopped` canaries occupy the canary slot.
- Model RPM / TPM / concurrency ceilings are counted uniformly across all entries.
- A prompt version freezes the four-piece set once referenced.
- Run results are immutable; annotations are stored in a separate table.
- A user token (`scope='user'`) is not bound to a single project, and a single token can be used at both the HTTP API and MCP entries simultaneously; in OSS it is unlimited in count, and there is no longer a "at most one valid Global MCP token at a time" constraint.
- Business resource queries must filter by `project_id`.
- A webhook token (`scope='webhook'`) must be bound to a `connector_id`; **the same connector supports multiple valid tokens coexisting in steady state** for per-consumer distribution; deleting the connector cascade-deletes all of its webhook tokens; after a token is revoked / expired, the row is **retained** (for `ph_runs.run_results.webhook_token_id` FK reference and audit traceability).

## 8. RLS and access boundary

The OSS edition does not rely on external Auth / RLS for business authentication. The Web UI runs as the local admin app by default; the HTTP API and MCP share the `scope='user'` user token; Webhook uses the per-connector `scope='webhook'` token.

Business tables do not create foreign keys to an external user table. Fields like `created_by` / `updated_by` / `submitted_by` / `locked_by` store an opaque actor UUID, used for local source tracking. Request entries are uniformly converted to an `ActorContext`, parsed by each entry resolver (`ActorContextResolver` / `McpAuthResolver` / `ConnectorContextResolver`); `ProjectContextResolver` performs the actor â†’ project access check at the HTTP / MCP entries, while the webhook entry does not go through this resolver. Access control is abstracted by `AccessControlService`, which passes everything by default in OSS and is replaced with an RBAC check in SaaS. See [08 Control-plane Adapter Boundary](08-saas-adapter-boundary.md).

## 9. Indexes and migrations

Indexes focus on high-frequency query paths:

- For backward compatibility with historically soft-deleted resources, `project_id + name` may continue to use a `deleted_at IS NULL` partial unique index. Active and archived prompt / dataset names remain reserved until permanent deletion physically removes the row.
- Run results are queried by `project_id + source + source_id + created_at`, `project_id + release_version_id + created_at`, `prompt_version_id + created_at`, `dbos_workflow_id`, and `bullmq_job_id`; webhook-entry attribution is aggregated by `(webhook_token_id, created_at) WHERE webhook_token_id IS NOT NULL`, used for per-consumer usage statistics.
- Experiments and optimizations are queried by `project_id + status` and creation time.
- Release lines guarantee release name uniqueness by `project_id + name`, and are queried by `project_id + status`, `prompt_id`, `input_connector_id`, and update time.
- Release events are queried by `release_line_id + created_at`, `lane_type + status`, and `prompt_id + created_at`.

All schema changes must modify `packages/db/src/schema/` and run `pnpm db:generate` to generate a migration. Do not modify the database directly with `psql`.
